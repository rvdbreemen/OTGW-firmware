# OTGW-firmware v1.4.0-beta Release Notes
**Release date:** 2026-04-18
**Branch:** dev → main (beta)
**Compare:** [v1.3.5...v1.4.0](https://github.com/rvdbreemen/OTGW-firmware/compare/v1.3.5...v1.4.0)

## Overview

v1.4.0 is a minor-version release with three themes:

1. **A modern toolchain.** ESP8266 Arduino core moves from 2.7.4 to 3.1.2, AceTime from 2.0.1 to 4.1.0, TelnetStream to 1.3.0, and the firmware migrates from ESPTelnet/TelnetStream to the new SimpleTelnet library. LittleFS handling was adjusted for the new core and debug input is now driven via a callback.
2. **A rewritten MQTT Home Assistant discovery pipeline.** The synchronous "just-in-time" discovery path is replaced by an async, bitmap-driven drip publisher. The legacy template path and its PROGMEM blobs are gone; climate, number, sensor, and Dallas discovery all go through a single streaming API.
3. **ESP8266 memory and stability work.** Large staging buffers are eliminated, the TCP buffer copy is shortened, several PROGMEM pointer mismatches and a reboot loop were fixed, and MQTT gating was calmed down.

There are no breaking changes vs v1.3.5 for default ESP8266 builds. MQTT topics, REST API shape, and settings format are preserved.

This build is published as **v1.4.0-beta** because the discovery rewrite and core upgrade are both large, and we want broader testing in the wild before promoting the tag to stable.

## New features

### Toolchain and libraries
- **ESP8266 Arduino core 3.1.2** (was 2.7.4). Full toolchain/SDK refresh.
- **AceTime 4.1.0** (was 2.0.1). API updated accordingly.
- **SimpleTelnet migration.** Replaces ESPTelnet/TelnetStream with a maintained, callback-driven debug port. The welcome banner and debug keys are preserved.
- **TelnetStream 1.3.0** where still referenced.

### MQTT Home Assistant discovery rewrite (#547)
- **Async drip publisher.** A new bitmap (`MQTTautoCfgPendingMap[8]`) tracks "needs publishing" alongside the existing "already published" bitmap. The OT message loop and sensors only set the pending bit — they never block. A `loopMQTTDiscovery()` timer drains **one** pending ID every ~3 seconds, automatically slowing to 30 s under `HEAP_WARNING` / `HEAP_CRITICAL` and restoring to 3 s when heap recovers.
- **Streaming API.** Climate, number, sensor, and Dallas discovery are now generated by a compact streaming API. The legacy `MqttHaSpecialCfg` path and its PROGMEM template blobs have been removed.
- **Comprehensive icon heuristics** in one place instead of per-template specials.
- **Runtime Dallas discovery.** Dallas sensor addresses are known only at runtime, so they are streamed via `streamDallasSensorDiscovery` rather than stored in the static PROGMEM array. The stale sensor entry for ID 246 was removed from the PROGMEM table.
- **Force-discovery (`F`) path fixed.** The serial `F` key now clears the "done" bitmap and re-sends all discovery messages.
- **No more 60+ message burst** on HA restart — discovery is spread over time, with less heap pressure and lower broker load.

### Configurable HA device identity
- `settings.device.sManufacturer` and `settings.device.sModel` (defaults `NodoShop` / `OTGW`) drive the HA device registry block. Configurable from the Web UI, REST API, and `settings.ini`.
- `otgw-pic/designer = "Schelte Bron"` is now published separately on the PIC info topic, crediting the original hardware and PIC firmware designer.

### Scheduled nightly restart (opt-in)
- `settings.bNightlyRestart` (default **off**) and `settings.iRestartHour` (default `4`).
- Checked once per minute via AceTime local time, in the configured timezone.
- Only fires when NTP is synced **and** uptime > 1 hour, to prevent reboot loops.
- Exposed via the settings JSON (`nightlyrestart`, `nightlyrestarthour`), Web UI fields with tooltips, and the REST API whitelist.

### REST API additions
- **`GET /api/v2/sensors/status`** — current Dallas sensor readings as a snapshot.
- **Debug access log:** a single-line format per request with HTTP status code and handler timing, and the noisy `onNotFound` routing lines removed.

### Debug ergonomics
- **MQTT debug split:** the single MQTT flag is split into **key 3** (MQTT communication) and **key 5** (MQTT publish gating).
- **NTP telemetry (key 6):** optional NTP sync diagnostics, on by default, listed in the telnet welcome banner.
- Noisy `skip` lines on MQTT gating are suppressed; per-publish MQTT error logging is suppressed to prevent serial overrun.
- Per-second work in `_debugBOL()` is cached to reduce debug output overhead; `AddLog` no longer does `O(n²)` strlen scans.

### Tooling
- New `/flash` skill for automated build + flash during development.

> **Scope note.** SAT (Smart Autotune Thermostat), ESP32 support, and the PlatformIO build system have been moved to a separate **`2.0.0`** branch and are **not** part of v1.4.0. The v1.4.0 line stays focused on ESP8266 on the Arduino IDE toolchain; earlier documentation that announced those features under v1.4.0 was premature.

## Bug fixes

- **PROGMEM-as-RAM crash in autodiscovery (Exception 3)** — fixed and followed up with a PROGMEM pool linkage validation guard to catch similar issues at boot.
- **Reboot loop on first 1.4.0-beta builds** — fixed via a heap guard, a `bConnected` check, and a named constant for the threshold.
- **Serial overrun on MQTT disconnect** — per-publish MQTT error logging is now suppressed.
- **LittleFS file opens** — missing null-checks added in the web server path.
- **LittleFS image size mismatch** under ESP8266 core 3.x.
- **MQTT status gate logic** — fixed state transitions and reduced debug log pressure.
- **REST API debug** — one-line access log, response status code, broken sensor address log cleaned up.
- **Dallas sensor ID 246** — removed from the static PROGMEM discovery array (it was never a real MSGID).
- **Spammy bitmap debug lines** in `setMQTTConfigDone` removed.
- **TelnetStream / ESPTelnet API incompatibilities** resolved as part of the SimpleTelnet migration.

## Internal improvements

- **Removed**: `trimInPlace()`, `resetMQTTBufferSize()`, duplicate `sendMQTTStreaming` chunk logic, `getMQTTConfigPending()`, `clearMQTTConfigPending()`, hardcoded `"Schelte Bron"` / `"otgw-nodo"` PROGMEM constants, the legacy `MqttHaSpecialCfg` and unused PROGMEM templates. Net: hundreds of lines removed.
- **Simplified**: `getMQTTConfigDone` / `setMQTTConfigDone` are now single-line bit ops; `sendMQTTStreaming` was merged into `sendMQTT`.
- **DRY**: a single `buildDiscoveryContext()` helper replaces four duplicated call sites; `doAutoConfigureMsgid` is simplified and the drip path is faster.
- **Heap thresholds relaxed** for the streaming architecture (`HEAP_CRITICAL` 3072 → 2048, `HEAP_WARNING` 5120 → 4096, `HEAP_LOW` 8192 → 6144, `MQTT_DISCOVERY_HEAP_MIN` 8000 → 4000, `STREAM_HEAP_MIN` 8000 → 4000) — old thresholds were sized for the 1200-byte template renderer.
- **`hourChanged()`** helper added (matching the pattern of `minuteChanged()`), used by hourly MQTT-connect heap diagnostics showing free heap and max contiguous block at each step of the MQTT connect sequence.
- **`WiFiClient::setSync(true) + setNoDelay(true)`** — drops the TCP_SND_BUF copy (~1072 bytes) and lowers MQTT latency. Combined with streaming publishes, the per-connection data-plane RAM drops from ~4.8 KB to ~3.7 KB.
- **`sLine[1200]` and `topicBuf[200]`** staging buffers eliminated (TASK-276/277).
- **lwIP** already configured as Low Memory (`lm2f`), confirmed — no change needed.

## Breaking changes

No breaking changes for ESP8266 default builds vs v1.3.5:

- MQTT topics are unchanged.
- REST API contracts are unchanged (new endpoint added, nothing removed).
- Settings JSON adds `nightlyrestart`, `nightlyrestarthour`, `device.manufacturer`, `device.model` — all with defaults, so existing `settings.ini` files keep working.

## Upgrade notes

- **Flash both firmware and filesystem.** The discovery rewrite changes internal bitmaps and debug keys, and ESP8266 core 3.x needs the new LittleFS image.
- **HA users:** on first boot, discovery is re-published gradually over a few minutes rather than in one burst. Existing entities will re-appear; stale entities from older MQTT topic experiments can be removed once discovery settles.
- **Custom manufacturer/model:** if you have multi-hardware setups, set `device.manufacturer` and `device.model` before re-running discovery.
- **Nightly restart** is **off** by default. Enable it only if you want the extra heap-recovery safety margin.
- **Beta tag.** This is `v1.4.0-beta`. We want a couple of weeks of community testing before we promote to a stable `v1.4.0` tag. Please report issues on [GitHub](https://github.com/rvdbreemen/OTGW-firmware/issues) or on [Discord](https://discord.gg/zjW3ju7vGQ).

## Known limitations

- **SAT, ESP32 support, and PlatformIO** are **not** part of this release. They have been moved to the separate **`2.0.0`** branch and will be tracked there until they are ready to ship. v1.4.0 targets ESP8266 on the Arduino IDE toolchain only.
