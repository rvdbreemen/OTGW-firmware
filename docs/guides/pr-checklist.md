# PR Checklist

Pre-merge verification for OTGW-firmware changes. Every PR that modifies firmware, build pipeline, or user-facing behaviour MUST pass the checks below before merge. "Builds clean" is the lowest bar, not the finish line.

This checklist lives under `docs/guides/` so it is grep-able next to the backlog CLI guide. `CLAUDE.md` links here from the Doing tasks section.

---

## Baseline gates (every PR)

Run before requesting review:

- [ ] **Builds clean on ESP8266 and ESP32-S3**
  ```bash
  python build.py
  ```
  Both binaries must appear in `build/` with size within a sensible range of the previous release. Sudden +50 KB is a red flag — investigate.

- [ ] **`evaluate.py --quick` health unchanged or improved**
  ```bash
  python evaluate.py --quick
  ```
  Health score must be at or above the previous commit's score. PROGMEM / String-class / binary-safe-compare rules all pass (or the PR is an explicit ADR-080 gate addition that introduces new rules).

- [ ] **Stdlib unittest suites green**
  ```bash
  python tests/test_evaluate.py
  python tests/test_build.py
  ```
  Add new cases for any new helper you touch in `evaluate.py` or `build.py`.

- [ ] **No uncommitted changes to `version.h` / `version.hash`** that were auto-generated by a previous build and forgotten. Stage and commit them intentionally or reset.

## Change-type gates (apply when relevant)

### Frontend changes (`data/*.html`, `data/*.js`, `data/*.css`)

- [ ] Open the affected page in a real browser on a flashed device OR a local LittleFS preview (whichever is practical). Target: Chrome / Firefox / Safari latest + 2.
- [ ] **Accessibility checks** where relevant:
  - Tab through interactive elements; every control is reachable.
  - Enter / Space activate buttons and custom toggles.
  - Screen reader announces collapsed/expanded state where aria-expanded is used.
  - Color contrast ≥ 4.5:1 on both light and dark themes for normal text (3:1 for large/UI).
- [ ] **Gzip assets** (for large `.js`): run `python build.py` and confirm `.gz` siblings appear in `src/OTGW-firmware/data/`. Network tab in DevTools should show `Content-Encoding: gzip` on the served asset.
- [ ] Try-catch around every `JSON.parse` on `response.json()`; `.ok` checked before consuming body; `.catch()` on every async chain.

### MQTT / Home Assistant discovery changes

- [ ] Run the procedure in `docs/api/MQTT_DISCOVERY_CHECK.md` against a real broker.
- [ ] `mosquitto_sub -t 'homeassistant/#'` after a post-change reboot: every msgid in the baseline table still publishes. Disappearing topics are release blockers.
- [ ] `docs/api/MQTT.md` updated if any topic or payload shape changed.
- [ ] HA entity UI: confirm the changed entity appears correctly after HA's next discovery cycle.

### Authentication / CSRF / session changes

- [ ] From a real browser, issue a cross-origin POST to `/api/v2/settings` with valid credentials — must be rejected (CSRF blocks it).
- [ ] From the same-origin web UI, issue the same POST — must succeed.
- [ ] `collectHeaders()` in `FSexplorer.ino` still registers every header the auth layer reads (Origin, Referer, If-None-Match at minimum) — this was silently broken in 2026-04 and is the template for future header-based logic.

### OTA / flash path changes

- [ ] Filesystem OTA upload a known-broken payload (e.g. too-large file). OTA must fail cleanly with a readable error on telnet, **not** corrupt the PIC serial stream.
- [ ] Firmware OTA upload a working build. Boot, check `/` loads, MQTT reconnects, PIC link alive.
- [ ] `flash_esp.py` downloads the release, SHA256 passes if the release publishes `SHA256SUMS`.

### OpenTherm / PIC / OTDirect changes

- [ ] Telnet log shows OT frames at expected rate (no flood, no stall).
- [ ] PS=0 baseline publishes `otmessage` topic. PS=1 suppresses it but decoded value topics keep flowing. Auto-leave-PS triggers after ~10s stall.
- [ ] No `Serial.print` calls leaked into the PIC UART path (ESP8266). `evaluate.py` must still flag the pattern.

### Build pipeline / `build.py` changes

- [ ] `python build.py --help` lists every expected flag (covered by `tests/test_build.py`, but spot-check).
- [ ] `--verify-image`, `--verify-flash`, `--reproducible`, `--ccache`, `--manifest` continue to work on both ESP8266 and ESP32 if they were already supported.
- [ ] `prepare_gzip_assets` runs when the filesystem is built (both mklittlefs and PlatformIO paths).

### Settings / persistence changes

- [ ] Saved `settings.ini` from a previous release still loads cleanly (forward-compat).
- [ ] New settings have reasonable defaults + comments in the appropriate `<Component>types.h` file per ADR-079.
- [ ] `OTGWSettings` / `OTGWState` byte sizes are unchanged if the PR claims to be a refactor (use `sizeof` print one-liner in `verboseConfig()` if needed).

### ADR changes

- [ ] ADR Status line follows ADR-080 classification (binding, structural, historical, policy, or guideline-level).
- [ ] New pattern-level ADRs reference their `evaluate.py` check or `tests/` case. No check → label is `guideline-level`.
- [ ] CLAUDE.md ADR Guidelines list is updated if the new ADR should be visible to future contributors.

## Backlog hygiene

- [ ] Task's ACs are either all checked, or explicit `--append-notes` explain why any AC was deferred and whether a follow-up task exists.
- [ ] `--final-summary` written in English, suitable as PR description.
- [ ] Task status set to `Done` only when the PR is actually merge-ready on this checklist.

---

## After the PR lands

- [ ] Push to the appropriate branch (`dev` for ongoing work, feature branches for isolated experiments).
- [ ] If the change touches release bits: open a release-notes entry if the project conventions call for one.
- [ ] Close referenced GitHub issues / Discord threads if any.

---

## When to skip items

Every item has a rationale. Skipping one is fine when the PR does not touch the relevant area (e.g. a pure docs PR does not need the OTA smoke test). Skipping an item that **does** apply needs a written justification in the PR description, not silence.

The checklist is a gate, not a ritual. If an item feels pointless for your specific change, say so and move on. If you are unsure, err on the side of running the check.
