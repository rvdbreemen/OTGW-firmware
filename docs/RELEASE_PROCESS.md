# Release Process

This document describes the standard pre-release checklist and release steps for OTGW-firmware.

---

## Pre-release checklist

Run through every item below before creating a GitHub release.

### 1. Version strings

- [ ] `src/OTGW-firmware/version.h` тАФ `_VERSION_PRERELEASE` is **commented out** (no `-beta`, `-rc`, etc.)
- [ ] `version.h` lines `_SEMVER_FULL`, `_SEMVER_NOBUILD`, `_VERSION` contain no pre-release suffix (e.g. `1.2.0+<hash>`, not `1.2.0-beta+<hash>`)
- [ ] File header comments in all `.ino`, `.h`, `.js`, `.css`, `.html`, `.cfg` files show the correct version (CI auto-fixes these on push via `scripts/autoinc-semver.py` тАФ verify after the last CI run)

> **How to check:** `grep -r "beta\|rc\|-dev" src/OTGW-firmware/ --include="*.h" --include="*.ino"`

### 2. Documentation

- [ ] `RELEASE_NOTES_<version>.md` exists, is up to date, and contains no "beta", "rc", or "this branch" language
- [ ] `RELEASE_GITHUB_<version>.md` exists (used as the GitHub release body)
- [ ] `README.md` top banner reflects the stable release (no "Development Release" or "beta" label)
- [ ] `README.md` "What's New" section heading matches the release version (no "-beta")
- [ ] Release notes table in `README.md` has the correct version entry (no "-beta")
- [ ] Previous release sections in `README.md` are condensed to highlights-only

> **How to check:** `grep -n "beta\|Development Release\|new in this branch" README.md RELEASE_NOTES_<version>.md`

### 3. No debug / placeholder artifacts

- [ ] No `TODO`, `FIXME`, `HACK`, `WIP`, or `remove before release` markers in firmware source (`src/OTGW-firmware/`)
- [ ] No test/debug default values in `src/OTGW-firmware/data/settings.ini`
- [ ] Default settings are safe for end users (e.g. `MQTTenable: false`, no hardcoded test broker)
- [ ] No accidental scratch or temp files in the repo root

> **How to check:** `grep -rn "TODO\|FIXME\|HACK\|WIP" src/OTGW-firmware/ --include="*.ino" --include="*.h" --include="*.cpp"`

### 4. CI / GitHub Actions

- [ ] All CI checks pass on `main` (green build on last commit)
- [ ] No hardcoded pre-release version strings or branch names in `.github/workflows/`
- [ ] Release workflow (`.github/workflows/release.yml`) triggers on `release: published` тАФ no changes needed
- [ ] The build action (`autoinc-semver.py`) is called with the correct directory and `--filename version.h`

### 5. Breaking changes documented

- [ ] All breaking changes are listed in `RELEASE_NOTES_<version>.md` under "Breaking Changes"
- [ ] Breaking changes are called out in `RELEASE_GITHUB_<version>.md` (GitHub release body)
- [ ] `README.md` "Important warnings / breaking changes" section is updated for the new version

### 6. Git state

- [ ] Working tree is clean (`git status` shows no uncommitted changes)
- [ ] Branch is `main` and is up to date with remote
- [ ] No `-beta` or `-rc` tag exists for this version (would confuse GitHub Releases)

---

## Release steps

Once the checklist above is complete:

1. **Push final commit to `main`** тАФ CI will run `autoinc-semver.py`, increment the build number, and push back an updated `version.h` (with file header comments corrected).

2. **Verify the CI push** тАФ Confirm the CI commit ("CI: update version.h") lands on `main` and the build is green.

3. **Create the GitHub release:**
   - Tag: `v<version>` (e.g. `v1.2.0`) pointing at the latest `main` commit
   - Title: `v<version>`
   - Body: paste the contents of `RELEASE_GITHUB_<version>.md`
   - Mark as **latest release** (not pre-release)

4. **Release workflow fires automatically** тАФ `.github/workflows/release.yml` builds and attaches `.elf`, `.bin`, and `.littlefs.bin` artifacts to the release.

5. **Post-release:**
   - Verify artifacts are attached to the GitHub release
   - Verify the `fwversion` field in `GET /api/v2/device/info` on a flashed device shows the correct version (no `-beta`)
   - Announce on Discord

## Version numbering

This project follows [Semantic Versioning](https://semver.org/):

| Component | Meaning |
|-----------|---------|
| **Major** | Incompatible API or protocol changes |
| **Minor** | New backward-compatible features |
| **Patch** | Backward-compatible bug fixes |
| **Pre-release** | `beta` during development; commented out for stable releases |
| **Build** | Auto-incremented by CI on every push; not user-controlled |

Version strings are generated by `scripts/autoinc-semver.py` from `src/OTGW-firmware/version.h`. Do not edit the derived macros (`_SEMVER_FULL`, `_SEMVER_NOBUILD`, `_VERSION`) by hand тАФ they will be overwritten by the next CI run. Instead, edit only `_VERSION_MAJOR`, `_VERSION_MINOR`, `_VERSION_PATCH`, and `_VERSION_PRERELEASE`.

## File locations

| File | Purpose |
|------|---------|
| `src/OTGW-firmware/version.h` | Authoritative version тАФ edited manually for major/minor/patch and pre-release |
| `scripts/autoinc-semver.py` | Auto-increments build, updates timestamps, propagates version to all file headers |
| `RELEASE_NOTES_<version>.md` | Full detailed release notes (linked from README) |
| `RELEASE_GITHUB_<version>.md` | Concise release body pasted into the GitHub release UI |
| `README.md` | Project readme тАФ top banner, "What's New", release notes table |
