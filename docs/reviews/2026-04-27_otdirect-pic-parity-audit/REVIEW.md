# OTDirect PIC-Parity Audit

Date: 2026-04-27

Branch: `feature-dev-2.0.0-otgw32-esp32-sat-support`

Coding agent: `codex`

Scope: OTDirect command abstraction layer, with special attention to whether it simulates PIC command behavior closely enough for firmware-internal command users, MQTT/REST command users, and external PIC-compatible clients.

## Method

This audit used two independent checks.

1. Firmware command usage check: searched the firmware for two-letter OTGW commands generated by SAT, MQTT, REST, NTP/time sync, and queue plumbing, then followed whether those commands reach the PIC path or OTDirect path.
2. Ground-truth comparison: compared OTDirect behavior with `other-projects/otgw-6.6/gateway.asm`, treating the PIC firmware as canonical for command semantics. Local manuals and the OpenTherm v4.2 message-id reference were used to validate message IDs and units.

Additional cross-project references were used only as supporting evidence, not as ground truth:

- `other-projects/otmonitor-6.6/otmonitor.vfs/gui.tcl`
- `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl`
- `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp`
- `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp`
- `other-projects/OT-Thing-OTGW32/Firmware/include/otvalues.h`
- `other-projects/OT-Thing-Pkunfazir/Firmware/src/otcontrol.cpp`

No production code was changed in this audit. The only repository writes are this report and Backlog tasks.

## Summary

The OTDirect layer is close in structure to a PIC-compatible command facade, but several high-value command semantics are not PIC-compatible today. The most important issues are not cosmetic: some commands are ignored, some valid PIC values are repurposed, and one MQTT-exposed command can write the wrong OpenTherm message.

## Findings

### F1 - `MM=0` is incorrectly treated as clearing max modulation

Task: `TASK-436`

Severity: High

Confidence: High

Evidence:

- SAT intentionally sends `MM=0` during OPV calibration and minimum-modulation paths: `src/OTGW-firmware/SATcontrol.ino:262`, `src/OTGW-firmware/SATcontrol.ino:267`, `src/OTGW-firmware/SATcontrol.ino:300`, `src/OTGW-firmware/SATcontrol.ino:301`, `src/OTGW-firmware/SATcontrol.ino:3722`.
- OTDirect documents MsgID 14 as `MM=` but clears the write override when the parsed value is zero: `src/OTGW-firmware/OTDirect.ino:2120`, `src/OTGW-firmware/OTDirect.ino:2123`, `src/OTGW-firmware/OTDirect.ino:2127`.
- PIC ground truth accepts numeric `MM=0..100` and only clears on parse failure/non-numeric input: `other-projects/otgw-6.6/gateway.asm:4752`, `other-projects/otgw-6.6/gateway.asm:4754`, `other-projects/otgw-6.6/gateway.asm:4758`, `other-projects/otgw-6.6/gateway.asm:4764`.
- OpenTherm v4.2 defines MsgID 14 as maximum relative modulation level setting with range 0..100 percent: `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:285`.
- OT-Thing OTGW32 also identifies ID 14 as maximum relative modulation level setting: `other-projects/OT-Thing-OTGW32/Firmware/include/otvalues.h:130`.

Why this is an issue:

`MM=0` is a valid PIC command value and a valid OpenTherm value. On OTDirect, the same firmware command releases the override instead of sending `0x0000`. That means SAT code that tries to force minimum modulation can silently fail on OTGW32 while working on PIC hardware.

Solution direction:

Keep this surgical. Change only the `MM=` handler so numeric zero enqueues MsgID 14 data `0x0000`. Preserve an explicit release path using the closest PIC-compatible behavior available to the parser, or document a separate OTDirect-only release alias if necessary.

### F2 - SAT sends unsupported `TW=` instead of `SW=` for DHW setpoint

Task: `TASK-437`

Severity: High

Confidence: High

Evidence:

- SAT tracks this as a `TW=` transition and builds `TW=<setpoint>`: `src/OTGW-firmware/SATcontrol.ino:234`, `src/OTGW-firmware/SATcontrol.ino:3466`, `src/OTGW-firmware/SATcontrol.ino:3471`, `src/OTGW-firmware/SATcontrol.ino:3472`.
- OTDirect implements `SW=` for MsgID 56 and has no `TW=` branch: `src/OTGW-firmware/OTDirect.ino:135`, `src/OTGW-firmware/OTDirect.ino:2096`.
- PIC ground truth documents `SW=60.0` for domestic hot water setpoint and routes `SW` in the command table: `other-projects/otgw-6.6/gateway.asm:57`.
- Local manuals document `SW=60`: `docs/MANUAL.md:3092`, `docs/MANUAL.md:3620`, `docs/manuals/en/ch09-api-reference.md:1010`.
- OpenTherm v4.2 defines MsgID 56 as `TdhwSet`: `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:252`.
- OTmonitor's DHW setpoint UI sends `SW=$dhwsetpoint`: `other-projects/otmonitor-6.6/otmonitor.vfs/gui.tcl:1132`, `other-projects/otmonitor-6.6/otmonitor.vfs/gui.tcl:1138`.
- OT-Thing OTGW32 routes its DHW setpoint MQTT command to `setDhwTemp`: `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp:212`, `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp:214`.

Why this is an issue:

`TW=` is not a supported PIC command and is not supported by OTDirect. The path will produce `NG` on OTDirect and would not be accepted by the PIC command table either. This is a clear spelling/command selection bug in SAT.

Solution direction:

Replace `TW=` with `SW=` at the SAT call site and update the matching log/comment. Do not add a `TW=` alias unless a real external compatibility requirement is found.

### F3 - OTDirect repurposes `GW=0` from monitor mode to bypass mode

Task: `TASK-438`

Severity: High

Confidence: High

Evidence:

- PIC ground truth says `GW=1` switches between monitor `(0)` and gateway `(1)` mode, and `GW=R` resets: `other-projects/otgw-6.6/gateway.asm:139`, `other-projects/otgw-6.6/gateway.asm:4687`.
- OTDirect maps `GW=0` to bypass relay mode, with rejection if no relay exists: `src/OTGW-firmware/OTDirect.ino:2344`, `src/OTGW-firmware/OTDirect.ino:2348`, `src/OTGW-firmware/OTDirect.ino:2355`.
- REST currently maps `mode=bypass` to `GW=0` and `mode=monitor` to `GW=M`: `src/OTGW-firmware/restAPI.ino:372`, `src/OTGW-firmware/restAPI.ino:373`, `src/OTGW-firmware/restAPI.ino:374`.
- Local manual tables still document `GW=0` as monitor mode: `docs/MANUAL.md:2500`, `docs/MANUAL.md:3090`, `docs/MANUAL.md:3636`, `docs/manuals/en/ch09-api-reference.md:333`, `docs/manuals/en/ch09-api-reference.md:1008`.
- MQTT docs contradict that by calling `GW=0` bypass: `docs/api/MQTT.md:587`.
- OTmonitor's gateway/monitor buttons send `GW=1` and `GW=0`: `other-projects/otmonitor-6.6/otmonitor.vfs/gui.tcl:1181`, `other-projects/otmonitor-6.6/otmonitor.vfs/gui.tcl:1183`.
- OT-Thing OTGW32 has bypass as a dedicated direct-control operation: `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp:381`.
- OT-Thing Pkunfazir also treats bypass as a separate relay/control state: `other-projects/OT-Thing-Pkunfazir/Firmware/src/otcontrol.cpp:245`, `other-projects/OT-Thing-Pkunfazir/Firmware/src/otcontrol.cpp:260`.

Why this is an issue:

`GW=0` is an established PIC command value. External clients, docs, and users reasonably expect it to select monitor mode. On OTDirect, the same command can physically bypass the gateway relay path or fail with `NG` on boards without a relay.

Solution direction:

Restore `GW=0` to monitor mode and keep `GW=1` gateway plus `GW=R` reset. Add an explicit OTDirect-only bypass alias such as `GW=P`, then update REST and docs. Avoid broad mode refactors.

### F4 - Generic MQTT command topics are ignored on OTDirect

Task: `TASK-439`

Severity: High

Confidence: High

Evidence:

- Runtime helpers separate PIC availability from command-interface availability: `isPICEnabled()` at `src/OTGW-firmware/OTGW-firmware.h:353`; `hasOTCommandInterface()` returns PIC or OTDirect at `src/OTGW-firmware/OTGW-firmware.h:370`, `src/OTGW-firmware/OTGW-firmware.h:371`.
- `addCommandToQueue()` routes directly to `handleOTDirectCommand()` when no PIC is present and OTDirect is enabled: `src/OTGW-firmware/OTGW-Core.ino:2849`, `src/OTGW-firmware/OTGW-Core.ino:2850`, `src/OTGW-firmware/OTGW-Core.ino:2854`.
- REST raw command handling already uses `hasOTCommandInterface()`: `src/OTGW-firmware/restAPI.ino:190`, `src/OTGW-firmware/restAPI.ino:212`.
- MQTT maps normal command topics to OTGW two-letter commands: `src/OTGW-firmware/MQTTstuff.ino:480`, `src/OTGW-firmware/MQTTstuff.ino:481`, `src/OTGW-firmware/MQTTstuff.ino:482`, `src/OTGW-firmware/MQTTstuff.ino:484`, `src/OTGW-firmware/MQTTstuff.ino:485`.
- MQTT then returns early when `!isPICEnabled()`, before sending raw or mapped commands to `addCommandToQueue()`: `src/OTGW-firmware/MQTTstuff.ino:882`, `src/OTGW-firmware/MQTTstuff.ino:895`, `src/OTGW-firmware/MQTTstuff.ino:905`.
- MQTT docs say commands are processed when an OT command interface, PIC or OTDirect, is detected: `docs/manuals/en/ch09-api-reference.md:1018`. The command table includes `command`, `setpoint`, `constant`, `hotwater`, `gatewaymode`, `maxdhwsetpt`, and `ctrlsetpt`: `docs/api/MQTT.md:551` through `docs/api/MQTT.md:562`.
- OT-Thing OTGW32 routes MQTT command topics directly into OpenTherm control without PIC hardware: `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp:212`, `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp:214`, `other-projects/OT-Thing-OTGW32/Firmware/src/mqtt.cpp:262`.

Why this is an issue:

The abstraction layer exists, and REST uses it, but MQTT still treats generic OTGW commands as PIC-only. OTGW32 users will see documented command topics silently ignored. This is likely to hide or worsen other parity bugs because MQTT automation never reaches OTDirect.

Solution direction:

Change the generic command-topic gate from PIC-only to command-interface availability. Keep PIC-only firmware flashing, PIC settings, and PIC diagnostics behind `isPICEnabled()`. Implement after `GW=0` and `TC=` semantics are made safer, because MQTT will expose those paths more broadly.

### F5 - `MI=` and `PR=N` report the wrong units on OTDirect

Task: `TASK-440`

Severity: Medium

Confidence: High

Evidence:

- PIC stores message interval internally in 5 ms units but prints milliseconds. `SetInterval` validates/stores, then jumps to `PrintInterval`: `other-projects/otgw-6.6/gateway.asm:4535`, `other-projects/otgw-6.6/gateway.asm:4564`, `other-projects/otgw-6.6/gateway.asm:4568`, `other-projects/otgw-6.6/gateway.asm:4573`. `PrintInterval` reconstructs decimal milliseconds with a final `0` or `5`: `other-projects/otgw-6.6/gateway.asm:2234`, `other-projects/otgw-6.6/gateway.asm:2237`, `other-projects/otgw-6.6/gateway.asm:2239`.
- OTDirect accepts milliseconds but responds with `val / 10`: `src/OTGW-firmware/OTDirect.ino:2743`, `src/OTGW-firmware/OTDirect.ino:2747`, `src/OTGW-firmware/OTDirect.ino:2749`, `src/OTGW-firmware/OTDirect.ino:2750`.
- OTDirect `PR=N` also reports `otMinIntervalMs / 10`: `src/OTGW-firmware/OTDirect.ino:2499`, `src/OTGW-firmware/OTDirect.ino:2500`.
- SAT sends `MI=500` as a real command: `src/OTGW-firmware/SATcontrol.ino:2888`, `src/OTGW-firmware/SATcontrol.ino:2890`.
- MQTT docs currently encode the OTDirect behavior rather than PIC ground truth, saying the acknowledgement is centiseconds: `docs/api/MQTT.md:577`, `docs/api/MQTT.md:653`.

Why this is an issue:

The scheduling value is stored as milliseconds, but clients see a response scaled down by 10. That breaks PIC-compatible command responses and makes `MI=500` look like `50`.

Solution direction:

Report milliseconds in `MI:` responses and `PR=N`. Do not change the scheduling model unless separate testing proves it is wrong.

### F6 - `SR=` parser is not PIC-compatible, and time/date sync is PIC-gated

Task: `TASK-441`

Severity: High

Confidence: High

Evidence:

- PIC command docs specify decimal byte syntax, for example `SR=18:1,230`: `other-projects/otgw-6.6/gateway.asm:123`.
- PIC `SetResponse` parses decimal DataID, colon, decimal byte, optional comma, and second decimal byte: `other-projects/otgw-6.6/gateway.asm:4771`, `other-projects/otgw-6.6/gateway.asm:4778`, `other-projects/otgw-6.6/gateway.asm:4782`, `other-projects/otgw-6.6/gateway.asm:4787`, `other-projects/otgw-6.6/gateway.asm:4789`, `other-projects/otgw-6.6/gateway.asm:4793`, `other-projects/otgw-6.6/gateway.asm:4799`.
- OTDirect documents and parses a single hex word: `src/OTGW-firmware/OTDirect.ino:2601`, `src/OTGW-firmware/OTDirect.ino:2605`.
- Firmware time sync builds PIC-style `SC`, `SR=21:month,day`, and `SR=22:yearHi,yearLow`, but returns unless PIC is available: `src/OTGW-firmware/networkStuff.ino:585`, `src/OTGW-firmware/networkStuff.ino:590`, `src/OTGW-firmware/networkStuff.ino:603`, `src/OTGW-firmware/networkStuff.ino:608`, `src/OTGW-firmware/networkStuff.ino:614`.
- OTmonitor sends the same PIC-style time/date commands: `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:1976`, `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:1978`, `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:1986`.
- OpenTherm v4.2 defines MsgID 20 day-time, MsgID 21 date, and MsgID 22 year: `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:198`, `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:199`, `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:200`.
- Local manual also documents NTP sync sending `SC`, `SR=21`, and `SR=22`: `docs/MANUAL.md:1250`.

Why this is an issue:

If OTDirect time sync is simply enabled by changing the gate, OTDirect would parse `SR=21:4,27` as a single hex value and lose the day byte. The parser must be fixed before widening the sync path.

Solution direction:

Teach OTDirect the PIC decimal byte syntax and make that canonical. Then change time/date sync to use `hasOTCommandInterface()` so OTDirect can receive the same commands as PIC. Retain hex syntax only if backwards compatibility is required and unambiguous.

### F7 - `CS` and `C2` overrides do not expire like PIC heartbeat commands

Task: `TASK-442`

Severity: Medium

Confidence: Medium-high

Evidence:

- PIC defines heartbeat flags for `CS` and `C2`: `other-projects/otgw-6.6/gateway.asm:495`, `other-projects/otgw-6.6/gateway.asm:496`.
- PIC `CommandExpiry` invalidates `CS` and `C2` after the minute timer expires unless a heartbeat was seen or the value is below the special low-setpoint threshold: `other-projects/otgw-6.6/gateway.asm:1619`, `other-projects/otgw-6.6/gateway.asm:1624`, `other-projects/otgw-6.6/gateway.asm:1626`, `other-projects/otgw-6.6/gateway.asm:1629`, `other-projects/otgw-6.6/gateway.asm:1630`, `other-projects/otgw-6.6/gateway.asm:1632`, `other-projects/otgw-6.6/gateway.asm:1635`.
- PIC sets heartbeat flags when the commands are accepted: `other-projects/otgw-6.6/gateway.asm:4196`, `other-projects/otgw-6.6/gateway.asm:5239`.
- OTDirect `CS=` and `C2=` enqueue persistent write overrides and clear only on explicit zero: `src/OTGW-firmware/OTDirect.ino:2022`, `src/OTGW-firmware/OTDirect.ino:2026`, `src/OTGW-firmware/OTDirect.ino:2029`, `src/OTGW-firmware/OTDirect.ino:2077`, `src/OTGW-firmware/OTDirect.ino:2081`, `src/OTGW-firmware/OTDirect.ino:2084`.
- OTmonitor explicitly schedules a `CS=0` expiry after a little over a minute: `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:576`, `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:578`, `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:1997`, `other-projects/otmonitor-6.6/otmonitor.vfs/otmonitor.tcl:2004`.
- SAT sends `CS=` continuously during control: `src/OTGW-firmware/SATcontrol.ino:3523`, `src/OTGW-firmware/SATcontrol.ino:3751`, and has explicit `CS=0` safety releases: `src/OTGW-firmware/SATcontrol.ino:1428`, `src/OTGW-firmware/SATcontrol.ino:2865`, `src/OTGW-firmware/SATcontrol.ino:3434`.

Why this is an issue:

One-shot command paths can leave a stale flow/control override active indefinitely on OTDirect. PIC behavior expects either refresh or expiry. This is a safety-relevant mismatch, but it needs careful handling because SAT intentionally sends recurring `CS=` commands.

Solution direction:

Track last accepted `CS` and `C2` times and expire those overrides after about one minute without refresh, matching PIC behavior as closely as practical. Keep the change limited to `CS` and `C2`.

### F8 - `TC=` is treated as `CS=` and writes MsgID 1

Task: `TASK-443`

Severity: High

Confidence: High

Evidence:

- PIC command comments distinguish temporary/constant temperature setpoints from control setpoint override: `TT=` and `TC=` at `other-projects/otgw-6.6/gateway.asm:46`, `other-projects/otgw-6.6/gateway.asm:48`; `CS=` at `other-projects/otgw-6.6/gateway.asm:52`.
- PIC override code is built around remote override state rather than simply treating `TC` as control setpoint: `other-projects/otgw-6.6/gateway.asm:428`, `other-projects/otgw-6.6/gateway.asm:2932`, `other-projects/otgw-6.6/gateway.asm:4094`.
- OTDirect write table comments and command handler map `CS=/TC=` to MsgID 1 and `TT=` to MsgID 16: `src/OTGW-firmware/OTDirect.ino:128`, `src/OTGW-firmware/OTDirect.ino:132`, `src/OTGW-firmware/OTDirect.ino:2022`, `src/OTGW-firmware/OTDirect.ino:2034`, `src/OTGW-firmware/OTDirect.ino:2041`, `src/OTGW-firmware/OTDirect.ino:2046`, `src/OTGW-firmware/OTDirect.ino:2053`.
- MQTT maps `constant` to `TC`, so this is user-accessible through normal automation: `src/OTGW-firmware/MQTTstuff.ino:482`, `docs/api/MQTT.md:553`, `docs/manuals/en/ch09-api-reference.md:1005`.
- OpenTherm v4.2 defines MsgID 1 as `TSet`, MsgID 16 as `TrSet`, MsgID 9 as `TrOverride`, and MsgID 100 as remote override function: `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:194`, `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:284`, `docs/opentherm specification/OpenTherm-Protocol-Specification-v4.2-message-id-reference.md:289`.
- OT-Thing OTGW32 keeps room setpoint and flow/control concepts separate in its direct-control code: `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp:263`, `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp:264`, `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp:717`, `other-projects/OT-Thing-OTGW32/Firmware/src/otcontrol.cpp:852`.

Why this is an issue:

`TC=` is documented and expected as a constant room setpoint override. On OTDirect it writes boiler control setpoint MsgID 1, the same as `CS=`. That can make an automation intending room setpoint control directly alter flow/control setpoint.

Solution direction:

Minimum safe fix: stop `TC=` from writing MsgID 1. Then either map it to the same room-setpoint behavior as `TT=` or implement a fuller PIC-style remote override path. The task requires documenting the chosen minimal behavior with PIC and OpenTherm evidence.

## Deferred Or Non-Findings

### `PR=T` formatting difference

OTDirect reports tweaks as `PR: T=%d/%d`: `src/OTGW-firmware/OTDirect.ino:2456`. The PIC `PrintTweaks` path appears to print the two flags without a slash: `other-projects/otgw-6.6/gateway.asm:2279`. I did not create a task because the observed impact is low and no local firmware path or external client dependency was found that would misbehave because of this formatting difference.

If a future command-response cleanup is planned, include `PR=T` in the regression fixture from `TASK-444`.

### OTDirect extension modes

`GW=S`, `GW=M`, and `GW=L` are OTDirect extensions documented in the OTDirect code/docs. I did not flag them by themselves because they do not reuse the established PIC `0`, `1`, or `R` values. The issue is specifically that `GW=0` was repurposed.

### `SW=0` and other zero-valued setpoint clears

This audit found strong proof for `MM=0` being valid OpenTherm data and actively used by SAT. Similar zero-as-clear patterns exist for other commands, but I did not create broader tasks without the same level of evidence. Keep the future work focused on proven cases unless a command-specific PIC/spec/client trail justifies widening the change.
