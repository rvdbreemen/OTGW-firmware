## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
## Context

While verifying TASK-481 (WebUI master-topic filter), the PROGMEM Compliance gate in `evaluate.py` showed 27 baseline violations on `feature-dev-2.0.0-otgw32-esp32-sat-support` HEAD. Investigation found the gate has two systematic false-positive sources that have nothing to do with actual PROGMEM compliance.

### Bug 1: macro continuation lines treated as ordinary code

In `evaluate.py::check_progmem_compliance()` (around line 1748-1752), the line filter is:

```python
if stripped.startswith('//') or stripped.startswith('/*') or stripped.startswith('#define'):
    continue
```

This skips a line ONLY if it literally starts with `#define`. But multi-line `#define` macros use backslash continuations; those continuation lines do NOT start with `#define`. Example from `src/OTGW-firmware/OTGWLogMacros.h:17-20`:

```cpp
#define AddLogf(fmt, ...)   ({ if (ot_log_pos < (OT_LOG_BUFFER_SIZE - 1)) { \
                                 int _w = snprintf(ot_log_buffer + ot_log_pos, OT_LOG_BUFFER_SIZE - ot_log_pos, fmt, ##__VA_ARGS__); \
                                 if (_w > 0) { size_t _rem = OT_LOG_BUFFER_SIZE - 1 - ot_log_pos; ot_log_pos += ((size_t)_w < _rem) ? (size_t)_w : _rem; } \
                               } })
```

Line 18 contains `snprintf(` and is NOT a `#define` line per the current filter; it gets flagged. Same pattern hits the `AddLog`, `AddLogln`, `ClrLog` macros in the same file (the `_P` variant on line 22 is fine because the regex specifically excludes `snprintf_P`).

The macro is correct: it expands inline at the call site, so the call site's compliance is what matters. The macro definition itself is irrelevant to RAM placement.

### Bug 2: auto-generated `OTGW-firmware.ino.cpp` is scanned

In `evaluate.py::check_progmem_compliance()` (around line 1730-1732), the file glob:

```python
code_files = (list(src_dir.glob("*.ino")) +
              list(src_dir.glob("*.cpp")) +
              list(src_dir.glob("*.h")))
```

This `*.cpp` glob picks up `OTGW-firmware.ino.cpp`, which is generated by the PlatformIO Arduino preprocessor as a single concatenation of all `.ino` files. First lines of the generated file:

```
# 1 "C:\\Users\\rvdbr\\AppData\\Local\\Temp\\tmpai429d4l"
#include <Arduino.h>
# 1 "D:/Users/Robert/Documents/GitHub/RvdB/OTGW-firmware/src/OTGW-firmware/OTGW-firmware.ino"
```

Effects:
- All `.ino` violations are counted twice (once in source, once in concatenation).
- Violation counts vary unpredictably between builds (depends on whether the preprocessor ran since last commit).
- Counts diverge between developers depending on local build state.

Same issue almost certainly affects other gates that use the same `code_files` glob pattern; needs audit.

## Solution

### Bug 1 fix
Track macro-continuation state across lines. When a non-comment line ends with `\` (after stripping whitespace), the next line is a macro continuation and should be skipped along with the `#define` opener.

### Bug 2 fix
Exclude auto-generated `*.ino.cpp` files. The most robust filter:

```python
code_files = [
    f for f in (list(src_dir.glob("*.ino")) + list(src_dir.glob("*.cpp")) + list(src_dir.glob("*.h")))
    if not f.name.endswith(".ino.cpp")
]
```

Or equivalently: detect Arduino preprocessor output via the first-line `# N "..."` linemarker pattern.

Audit other gates in `evaluate.py` that use the same glob pattern; apply the same exclusion where appropriate.

## Out of scope

- Fixing the actual baseline violations in production code (separate task; a real-world cleanup that would now be measurable once these false positives are removed).
- Adding new PROGMEM patterns (e.g., `printf`, `Serial.print`); the current pattern set is fine.

## Summary

Fixed two systematic false-positive sources in evaluate.py's PROGMEM compliance gate (and other gates that share the same source-file glob).

**Bug 1 (macro continuation lines):** `check_progmem_compliance()` now tracks a `prev_was_continuation` state: lines following a backslash-terminated line (whether a `#define` opener, a comment, or another continuation line) are skipped, with the chain continuing as long as the current line also ends with a backslash. Eliminates the `OTGWLogMacros.h:18` false positive on the multi-line `AddLogf` macro body.

**Bug 2 (auto-generated `*.ino.cpp`):** Introduced module-level helper `collect_firmware_source_files(src_dir, *, include_ino, include_cpp, include_h)` that filters `*.cpp` files whose name ends with `.ino.cpp`. The PlatformIO Arduino preprocessor output is no longer scanned, so violation counts no longer inflate when a build state is present.

**Audit pass (AC #4):** applied the helper at every call site that previously used the same `*.ino + *.cpp [+ *.h]` glob pattern — 9 sites in total: ADR-086 time-boundary (514), ADR-062 reset audit (712), Serial.print check (1638), large buffers (1702), PROGMEM compliance (1730), ArduinoJson ban (1781), Stack safety (1820), Security analysis (2076), Binary-safe compare (2263). Two pure ino-only / h-only globs left intact (Path.glob('*.ino') does NOT match `*.ino.cpp`, verified empirically).
