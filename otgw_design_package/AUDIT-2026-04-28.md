## 1. Bevindingen uit de screenshots vs. evidence

Acht visuele klachten, teruggebracht tot **vier** onderliggende oorzaken. Per oorzaak: bewijs uit de code en hoe het op 1.4.1 wel goed stond.

### Oorzaak A вЂ” Brand-kleuren bewust gedesatureerd in `ds-tokens.css`

De auteur van het design-package heeft de cyan-identiteit opzettelijk weggehaald. Letterlijk commentaar in de tokens-file:

```
/* Brand / accent вЂ” desaturated. The live UI is much grayer than the
   original cyan suggested; brand color survives only as the faintest [..] */
--brand-cyan:        #cfe9ee;   /* Nav strip вЂ” pale washed cyan, almost gray */
--brand-cyan-soft:   #f1f7f8;   /* App background вЂ” barely-tinted off-white */
--brand-sky:         #b8c4c8;   /* hover accent вЂ” cool gray */
--brand-lightblue:   #d6d8da;   /* Buttons / data rows вЂ” neutral gray, not blue */
```
*(`ds-tokens.css:51-58`)*

Vergelijking met `1.4.1` (`git show 1.4.1:src/OTGW-firmware/data/index.css`):

| Element | 1.4.1 (intended) | huidige design-package | screenshot |
|---|---|---|---|
| `html` background | `#e6ffff` (licht cyaan) | `--brand-cyan-soft: #f1f7f8` (off-white) | Image #1: bijna wit ipv mintcyaan |
| `.nav-container` strip | `#00bffe` (helder cyaan) | `--brand-cyan: #cfe9ee` (vaal) | Image #1, #5, #6: grijs-cyan strip |
| `.nav-item` knoppen | `lightblue` (`#add8e6`) | `--brand-lightblue: #d6d8da` (grijs) | Image #1: grijze knoppen |
| `table / .otmonrow / .devinforow` | `background: lightblue` | (geen rule, default white) | Image #1, #6: rijen wit |

Ironisch: dark-mode in dezelfde file *heeft* wГ©l `#00bffe` voor `--brand-cyan` (`ds-tokens.css:144`). De auteur wist welke kleur "echt" was, maar koos voor light-mode een andere richting.

**Conclusie:** dit is een ontwerpkeuze van de package, geen bug. EГ©n regelvalwijziging in ds-tokens.css herstelt 80% van de visuele klacht.

### Oorzaak B вЂ” Legacy class-definities ontbreken in `components.css`

Het design-package liep een handoff-stap (Patch E in `handoff.md`): "remove legacy CSS files" (`index.css`, `index_dark.css`, `FSexplorer.css`). Maar de JavaScript-functies die `index.html` runtime vullen blijven HTML uitstoten met **legacy klassenamen**. Die namen hebben geen rule meer in `components.css` в†’ ongestylde stack van block-elementen.

Concreet missend in `components.css`:

| Klasse | GeГ«mit door | Locatie in oude `index.css` (1.4.1) |
|---|---|---|
| `.pictable`, `.picrow`, `.piccolumn1..5` | `refreshPICinfo()` (`index.js:3943-4130`) | `index.css:335-362` |
| `.devinforow`, `.devinfocolumn1`, `.devinfocolumn2` | `refreshDeviceInfo()` (`index.js:4813-4862`) | `index.css:456-475` |
| `.otmontable`, `.otmonrow`, `.otmoncolumn1..3` | hoofdpagina monitor | `index.css:419-450` |
| `.firmware-row-bold` | PIC firmware page header | `index.css:1077` |
| `.hidden` (globaal) | overal вЂ” toggleHidden() in index.js | `index.css:1025: .hidden { display: none !important }` |

De `.hidden`-regel ontbreken verklaart **direct** de open dropdown (Image #2): `<div class="adv_dropdown hidden">` werkt alleen als `.hidden { display: none }` ergens gedefinieerd staat. Components.css heeft de declaratie nooit overgenomen.

**Conclusie:** dit is een directe bug in het design-package: het verwijderde de oude CSS zonder elke benodigde regel te porteren. Geen ontwerpkeuze, maar een onvolledige migratie.

### Oorzaak C вЂ” Footer Save-knop is niet scoped per page-section

`index.html:554-558` plaatst de Save-knop in `<div class="footer">`, **buiten** alle `.page-section` containers. Hij is dus altijd zichtbaar wanneer JS de `hidden` klasse weghaalt. Het toggle-mechanisme draait via `toggleHidden('btnSaveSettings', hideOnly)` in `index.js:3129`.

Het probleem in Image #3 is dus dat **er code is die de Save-knop laat zien op de hoofdpagina**. De waarschijnlijke oorzaak: bij het wisselen van page-section wordt `btnSaveSettings` niet expliciet weer verborgen, dus blijft hij hangen in de zichtbare staat van de vorige pagina.

Op 1.4.1 gebeurde hetzelfde structureel; de UX werkte daar echter wel. Dat suggereert dat de design-package ГіГіk de show/hide-aansturing van de save-bar heeft veranderd (sticky save-bar uit DS:SETTINGS-SAVEBAR / TASK-461). Cross-check nodig: vergelijk de hide-logic tussen 1.4.1 en huidig.

**Conclusie:** combinatie van structurele keuze (footer-Save buiten sections) + verschoven toggle-logic. Twee mogelijke fixes: scope de Save-knop binnen de settings page-section, of zorg dat tab-switching de Save-knop expliciet verbergt.

### Oorzaak D вЂ” Schaal-/spacing-tokens te ruim voor dichte tabellen

Settings-rijen (Image #7), command-input (Image #4) en webhook-velden (Image #8) gebruiken design-tokens (`--space-2`, `--space-4`, etc.) die voor losse, ademende cards gekalibreerd zijn. Voor dichte settings-tables komt dat over als "te hoog / te veel padding".

| Element | Huidige rule | Resultaat |
|---|---|---|
| `.settingDiv` | `padding: var(--space-2) var(--space-5)` (8px 20px) + `border: 4px solid` | Rijen ~50px hoog |
| `.settingDiv input[type="text"]` | `width: 240px` *vast* | Webhook-URL afgekapt |
| `.ot-cmd-input` | erft generieke input padding (`var(--space-2) var(--space-4)`) | Veel hoger dan de Send-knop ernaast |

Op 1.4.1 was `.settingDiv` strakker (`padding: 2px ...`), input gebruikte `size="60"`-attribuut, en `.ot-cmd-input` had een eigen 32px-hoogte regel.

**Conclusie:** geen broken token-system, wel verkeerde toepassing van tokens op compacte componenten. Vereist component-specifieke overrides.

### Route 1 вЂ” **"Hard reset"**: design-package verwijderen, terug naar 1.4.1 CSS

### Route 2 вЂ” **"Token reset + class restore"**: design-package houden, kleuren rebranden, missende klassen porteren

**Wat:** drie gerichte edits in `data/`:

1. **`ds-tokens.css`** вЂ” overschrijf de `:root`-light-mode brand vars met 1.4.1 hex codes:
   ```
   --brand-cyan:        #00bffe;
   ```

3. **`components.css`** вЂ” knijp `.settingDiv` (padding `var(--space-1) var(--space-4)`, geen 4px border maar 1px), maak `.settingDiv input[type="text"]` responsief (`width: auto; min-width: 240px; max-width: 100%;`), geef `.ot-cmd-input` een `height: 32px`.

- вњ… **Pro:** behoudt alles wat goed is aan het package: SAT-pagina, dark mode, theme-toggle, /design.html. Token-driven, dus toekomstige rebranding is centraal.
- вњ… **Pro:** dark-mode wordt **automatisch** ook helderder cyaan zonder extra werk (de dark `--brand-cyan` was al `#00bffe`).
- вќЊ **Con:** meer werk dan Route 1 вЂ” elke missende klasse moet je inhoudelijk doorlopen en testen. Risico: je vergeet een klasse en een andere pagina blijkt later ook stuk.
- рџ•’ **Inzet:** ~6-8 uur over 2-3 commits (1 per cluster: tokens, legacy widgets, density tweaks).
- рџ”Ѓ **Risico:** middel. Iteratief; per commit kun je flashen en visueel valideren.

**Wanneer wel:** je wilt de design-system investering vasthouden en bent bereid de migratie *echt af te maken*.

### Route 3 вЂ” **"Hybride"**: legacy index.css naast de design-package

**Wat:** zet `index.css` van 1.4.1 terug, link 'm in `index.html` *na* `components.css`. Behoud `ds-tokens.css` + `components.css` voor SAT/theme-toggle. CSS-cascade laat de legacy regels winnen (laatste binnenkomende selector wint bij gelijke specificiteit).

- вњ… **Pro:** snelste pad terug naar "ziet er goed uit". Behoudt zowel SAT-styling als legacy widgets.
- вќЊ **Con:** twee concurrerende CSS-systemen. Toekomstige wijzigingen vereisen kennis van beide. Tokens en hex-codes leven naast elkaar.
- вќЊ **Con:** dark-mode wordt kapot gemaakt вЂ” legacy `index_dark.css` ging via een aparte stylesheet, design-package gebruikt `[data-theme="dark"]` op root. Beide systemen tegelijk laten draaien is een doos van Pandora.
- рџ•’ **Inzet:** ~3-4 uur (link toevoegen, dark-mode reconciliatie, conflict-debugging).
- рџ”Ѓ **Risico:** middel-hoog. Werkt vandaag, breekt morgen wanneer iemand iets in components.css aanpast en niet in legacy index.css.

**Wanneer wel:** je hebt een release-deadline en wilt vandaag een werkende UI; lange-termijn opruimen accepteer je later.
